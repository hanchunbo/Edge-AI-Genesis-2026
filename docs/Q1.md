## 第一阶段：现代 C++ 核心内存与性能优化 (W1-W5)

### W1：内存安全、RAII 机制与智能指针

**学习目的**  
AI 推理程序常驻运行，内存泄漏是 OOM 崩溃的元凶。掌握 RAII 是从"写逻辑"到"写系统"的转变。

**核心范围**
- 智能指针（`unique_ptr` / `shared_ptr` / `weak_ptr`）底层原理
- 所有权转移逻辑与生命周期管理
- 析构函数闭环与异常安全
- `weak_ptr` 解决循环引用问题

**实战作业**  
编写 `SafeTensorBuffer` 类管理模拟显存/内存块：
- 构造函数分配资源
- 禁用拷贝构造（`= delete`）
- 支持移动语义
- 在抛出异常时验证资源 100% 自动回收

**成功指标**
- `Valgrind` 检测报告显示 "0 bytes leaked"
- 能准确说出 `shared_ptr` 引用计数原子操作对高频推理的性能损耗

**产出物**: `01_Linux_CPP_Foundations/memory_safety/safe_tensor_buffer.hpp`

---

### W2：零拷贝思想：移动语义与右值引用

**学习目的**  
在边缘端（如 3060 或 VPS）处理 4K/8K 图像时，内存拷贝是巨大的性能浪费。

**核心范围**
- 右值引用 (`T&&`) 本质理解
- `std::move` 的实现原理
- 移动构造函数与移动赋值运算符编写
- 完美转发 `std::forward`

**实战作业**  
编写性能 Benchmark 脚本，对比将 1000 帧 4K 图像（`CustomImage` 类）存入 `std::vector` 时深度拷贝与移动语义的耗时差异。

**成功指标**
- 移动操作耗时应接近 0ms 且不随数据量增加而增长
- 通过 `static_assert` 在编译期验证类型的可移动性

**产出物**: `01_Linux_CPP_Foundations/move_semantics/benchmark_move_vs_copy.cpp`

---

### W3：C++17 跨平台实战特性与视图优化

**学习目的**  
规范模型文件管理逻辑，利用视图技术减少不必要的字符串内存分配。

**核心范围**
- `std::filesystem` 跨平台路径管理
- `std::optional` 优雅处理异常返回
- `std::string_view` 视图优化
- 结构化绑定 (Structured Bindings)

**实战作业**  
编写 "AI 模型自动扫描器"：
- 递归扫描 `/models` 目录
- 筛选 `.onnx` / `.engine` 文件
- 使用 `optional` 返回包含文件名与大小的元数据
- 解析逻辑强制使用 `string_view`

**成功指标**
- 核心解析代码无冗余的 `std::string` 临时对象产生
- 代码在 Linux/Windows 下一键编译且路径处理逻辑一致

**产出物**: `01_Linux_CPP_Foundations/filesystem/model_scanner.cpp`

### W4：并发编程基础：多线程与任务同步

**学习目的**  
利用多核架构加速预处理。理解"预处理-推理-后处理"流水线的并行逻辑。

**核心范围**
- `std::thread` 生命周期管理
- `std::mutex` 与 `std::lock_guard`
- 条件变量 `std::condition_variable`
- 数据竞争检测与调试

**实战作业**  
实现 "生产者-消费者" 图像处理原型：
- 线程 A 模拟图像采集入队
- 线程 B 模拟预处理出队
- 实现线程安全的循环队列 (Thread-safe Ring Buffer)

**成功指标**
- 系统在高并发下不出现死锁（Deadlock）
- 测算处理延迟，验证 B 线程能及时响应 A 线程的通知
- `ThreadSanitizer` 检测无数据竞争

**产出物**: `01_Linux_CPP_Foundations/threading/producer_consumer.cpp`

### W5：高性能并发进阶：通用线程池架构

**学习目的**  
频繁创建/销毁线程开销巨大，线程池是多路视频流 AI 推理的工业标配。

**核心范围**
- 基于 `std::queue` 的任务队列设计
- 使用 `std::future` 和 `std::packaged_task` 实现异步结果获取
- 泛型任务提交（模板编程）
- 线程池优雅关闭策略

**实战作业**  
手写支持泛型任务提交的 `ThreadPool` 类：
- 支持向池中提交任意 Lambda 函数
- 获取异步执行结果
- 实现工作窃取（Work Stealing）优化

**成功指标**
- 成功并行执行 100 个模拟图像旋转任务
- 线程池在任务空闲时处于低功耗挂起状态（CPU 占用趋近 0%）

**产出物**: `01_Linux_CPP_Foundations/thread_pool/thread_pool.hpp`

## 第二阶段：Linux 系统工程与高性能构建 (W6-W8)

### W6：Linux 高性能 I/O：文件映射 mmap

**学习目的**  
加载数 GB 模型权重时，`mmap` 是零拷贝加载、提升启动速度的关键。

**核心范围**
- `mmap` 原理与系统调用
- 系统调用 `open`/`read` 与标准库缓冲对比
- 内存页对齐基础
- 虚拟内存与物理内存映射

**实战作业**  
对比实验：编写程序读取 500MB 权重文件：
- 记录使用 `fstream` 的加载耗时
- 记录使用 `mmap` 的加载耗时
- 分析页缓存 (Page Cache) 对性能的影响

**成功指标**
- `mmap` 版加载速度明显优于传统 I/O
- 能解释内存映射在进程间通信（IPC）中的潜在应用

**产出物**: `01_Linux_CPP_Foundations/mmap_loader/mmap_benchmark.cpp`

### W13: 阶段性项目

**项目名称**: 高性能多线程图像预处理引擎

**任务描述**  
综合 Q1 所有知识，实现一个基于线程池的图像处理系统。

**核心要求**
1. 使用 CMake 构建，集成 GTest
2. 支持多路视频/图像流并发处理
3. 核心算子（Resize/Gray）采用 W9-W10 中手写的优化版本
4. 输出详尽的各模块耗时报告（为 Q2 数据分析预热）

**系统架构**

```
Input Stream ──► Decode ──► [ThreadPool] ──► Processing Pipeline ──► Output
                               │
                   ┌───────────┼───────────┐
                   ▼           ▼           ▼
               Worker 0    Worker 1    Worker N
                   │           │           │
                   └───────────┴───────────┘
                               │
                   ┌───────────┴───────────┐
                   ▼           ▼           ▼
               BGR2Gray    Resize     Letterbox
```

**成功指标**  
产出一套可复用的 C++ 部署框架原型，作为 2026 简历的首个硬核项目。
